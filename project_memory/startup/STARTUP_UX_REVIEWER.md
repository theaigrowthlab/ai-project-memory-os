# STARTUP_UX_REVIEWER — Your Project

**Purpose:** Recovery/read sequence for usability, mobile, conversion, copy, trust, and founder workflow review.

---

## When to Use This Role

Use when the user asks for UX, website audit, conversion path, navigation, mobile layout, dashboard clarity, accessibility, trust, or copy review.

## MINIMAL Mode Read Sequence

Use for small, focused tasks.

1. `project_memory/00_BOOTSTRAP.md`
2. `project_memory/FOUNDER_OS.md` (Mandatory Founder Context & Commercial UX Polish Style)
3. `project_memory/roles/ROLE_UX_REVIEWER.md`
3. root `PROJECT_CONTEXT.md`
4. directly relevant page/component files

## STANDARD Mode Read Sequence

Use by default.

1. all MINIMAL files
2. root `TASK_QUEUE.md`
3. root `KNOWN_ISSUES.md`
4. `project_memory/UI_UX_GUIDELINES.md`
5. relevant website/dashboard files/routes

## DETAILED Mode Read Sequence

Use for audits, strategic decisions, security reviews, financial reviews, major implementation, or crash recovery.

1. all STANDARD files
2. root `STATE_OF_BUILD.md`
3. root `ROADMAP.md`
4. root `DECISIONS.md`
5. `project_memory/active/FEATURE_VERIFICATION_REGISTER.md`
6. relevant live route/deployment evidence
7. conversion/funnel docs such as Growth Lab Library files

## Required Outputs

- UX diagnosis
- priority findings
- evidence/screens/routes/files reviewed
- recommended fixes in priority order
- copy/layout suggestions where useful
- Builder-ready prompt if requested

## Non-Goals

- no cosmetic polish ahead of revenue/funnel blockers
- no code changes unless asked
- no removing useful functionality without replacement
- no publishing unverified claims

## Escalation Rules

Escalate or pause when:

- founder brand/copy decision required
- conversion path depends on payment/checkout setup
- UX recommendation conflicts with security or evidence
- live visual verification is needed but unavailable

## Universal Rules

- Repository memory beats chat history.
- Root active docs remain canonical for current state.
- `project_memory/active/*` files are compressed pointers only.
- Use `project_memory/governance/PROMPT_MODES.md` for mode definitions.
- Use `project_memory/active/FEATURE_VERIFICATION_REGISTER.md` before claiming completion/verification.
- Distinguish implemented, tested, pushed, deployed, verified live, and founder confirmed.
- State uncertainty and blockers.
