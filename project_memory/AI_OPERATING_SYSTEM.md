## 4. Development Philosophy

- Prefer incremental, tested improvements.
- Avoid large rewrites unless unavoidable.
- No new agents/features unless directly tied to revenue, lead generation, retention, security, or founder efficiency.
- Keep mock/demo behaviour clearly labelled.
- Preserve backward compatibility where practical.
- Do not remove working functionality without reason and documentation.

---

## 5. Testing Requirements

Before claiming complete:

- run relevant tests,
- run the full suite when core engine files change,
- add regression tests for bugs/security fixes,
- verify render if UI changed,
- verify deployment if claiming live,
- record exact evidence.

---

## 6. Deployment Requirements

Always distinguish:

- Designed, not implemented
- Implemented, not tested
- Tested locally
- Pushed
- Deployed, not verified
- Verified live
- Founder confirmed
- Blocked

### Mandatory Git identity for [Your Frontend Hosting]-connected repos

Before committing to any repo connected to [Your Frontend Hosting], configure:

```bash
git config user.name "your-project"
git config user.email "info@your-project.co.uk"
```

Do **not** use `builder@arena.ai` for [Your Frontend Hosting]-triggering commits. [Your Frontend Hosting] blocks deployments when the commit email cannot be matched to a GitHub account.

## 7. Communication Standards

- Be direct.
- State uncertainty.
- Provide evidence.
- Do not bury blockers.
- Do not fabricate progress.
- Use founder-friendly language.
- Explain what changed and why in plain English.

---

## 8. Decision Hierarchy

1. Founder instruction
2. Security/safety constraints
3. Revenue roadmap
4. Project memory canonical docs
5. Tests/evidence
6. AI judgement

---

## 9. Prohibited Behaviour

- Do not expose secrets.
- Do not invent live verification.
- Do not duplicate project memory.
- Do not add feature sprawl.
- Do not silently alter strategic direction.
- Do not remove working functionality without reason.
- Do not claim revenue/sales without evidence.
- Do not enable payment CTAs before real checkout exists.

---

## 10. Gamification Guardrail

Your Project is an AI Business Operating System, not a business-themed game. Gamification, agent identity, HQ, Office, missions, XP, streaks, achievements, and future companion experiences are allowed only when they support at least one of:

- revenue generation,
- founder efficiency,
- business visibility,
- decision support,
- risk reduction,
- retention.

Do not prioritise standalone games, deep RPG systems, arbitrary XP, cosmetic simulation loops, inventory/loot systems, or decorative office mechanics while revenue systems are immature.
